12 /*
13  SegmentedVector is a vector-like class with the following advantages over
14  vector:
15  1. Resizing has no memory spike. (*)
16  2. Should work more nicely with fragmented memory because data is
17  partitioned into fixed-size chunks of size SEGMENT_BYTES.
18  3. Overallocation is only additive (by SEGMENT_BYTES), not multiplicative
19  as in vector. (*)
20  4. References stay stable forever, so there is no need to be careful about
21  invalidating references upon growing the vector.
22 
23  (*) Assumes that the size of the "segments" vector can be neglected, which is
24  true if SEGMENT_BYTES isn't chosen too small. For example, with 1 GB of data
25  and SEGMENT_BYTES = 8192, we can have 131072 segments.
26 
27  The main disadvantage to vector is that there is an additional indirection
28  for each lookup, but we hope that the first lookup will usually hit the cache.
29  The implementation is basically identical to that of deque (at least the
30  g++ version), but with the advantage that we can control SEGMENT_BYTES. A
31  test on all optimal planning instances with several planner configurations
32  showed a modest advantage over deque.
33 
34  The class can also be used as a simple "memory pool" to reduce allocation
35  costs (time and memory) when allocating many objects of the same type.
36 
37  SegmentedArrayVector is a similar class that can be used for compactly
38  storing many fixed-size arrays. It's essentially a variant of SegmentedVector
39  where the size of the stored data is only known at runtime, not at compile
40  time.
41 */
