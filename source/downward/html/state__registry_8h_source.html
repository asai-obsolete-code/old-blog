15 /*
16  Overview of classes relevant to storing and working with registered states.
17 
18  GlobalState
19  This class is used for manipulating states.
20  It contains the (uncompressed) variable values for fast access by the heuristic.
21  A State is always registered in a StateRegistry and has a valid ID.
22  States can be constructed from a StateRegistry by factory methods for the
23  initial state and successor states.
24  They never own the actual state data which is borrowed from the StateRegistry
25  that created them.
26 
27  StateID
28  StateIDs identify states within a state registry.
29  If the registry is known, the ID is sufficient to look up the state, which
30  is why ids are intended for long term storage (e.g. in open lists).
31  Internally, a StateID is just an integer, so it is cheap to store and copy.
32 
33  PackedStateBin (currently the same as unsigned int)
34  The actual state data is internally represented as a PackedStateBin array.
35  Each PackedStateBin can contain the values of multiple variables.
36  To minimize allocation overhead, the implementation stores the data of many
37  such states in a single large array (see SegmentedArrayVector).
38  PackedStateBin arrays are never manipulated directly but through
39  a global IntPacker object.
40 
41  -------------
42 
43  StateRegistry
44  The StateRegistry allows to create states giving them an ID. IDs from
45  different state registries must not be mixed.
46  The StateRegistry also stores the actual state data in a memory friendly way.
47  It uses the following class:
48 
49  SegmentedArrayVector<PackedStateBin>
50  This class is used to store the actual (packed) state data for all states
51  while avoiding dynamically allocating each state individually.
52  The index within this vector corresponds to the ID of the state.
53 
54  PerStateInformation<T>
55  Associates a value of type T with every state in a given StateRegistry.
56  Can be thought of as a very compactly implemented map from GlobalState to T.
57  References stay valid forever. Memory usage is essentially the same as a
58  vector<T> whose size is the number of states in the registry.
59 
60

62  Usage example 1
63  ---------------
64  Problem:
65  A search node contains a state together with some information about how this
66  state was reached and the status of the node. The state data is already
67  stored and should not be duplicated. Open lists should in theory store search
68  nodes but we want to keep the amount of data stored in the open list to a
69  minimum.
70 
71  Solution:
72 
73  SearchNodeInfo
74  Remaining part of a search node besides the state that needs to be stored.
75 
76  SearchNode
77  A SearchNode combines a StateID, a reference to a SearchNodeInfo and
78  OperatorCost. It is generated for easier access and not intended for long
79  term storage. The state data is only stored once an can be accessed
80  through the StateID.
81 
82  SearchSpace
83  The SearchSpace uses PerStateInformation<SearchNodeInfo> to map StateIDs to
84  SearchNodeInfos. The open lists only have to store StateIDs which can be
85  used to look up a search node in the SearchSpace on demand.
86 
87  ---------------
88  Usage example 2
89  ---------------
90  Problem:
91  In the LMcount heuristic each state should store which landmarks are
92  already reached when this state is reached. This should only require
93  additional memory, when the LMcount heuristic is used.
94 
95  Solution:
96  The heuristic object uses a field of type PerStateInformation<std::vector<bool> >
97  to store for each state and each landmark whether it was reached in this state.
